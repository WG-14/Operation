You are working in the GitHub repository `WG-14/bithumb-bot`. Your task is to verify and fix the research/runtime strategy-equivalence risk without weakening the core concern or reducing it to superficial patches. This task is based on the prior repository review and risk analysis: 

## Core problem to address

Do not treat the current diagnosis as proven. Treat it as a set of hypotheses to validate or falsify in the repo.

The suspected structural issue is that SMA strategy semantics are not fixed as one executable contract. Instead, they appear to be distributed across:

* runtime strategy logic: `src/bithumb_bot/strategy/sma.py`
* runtime exit rules: `src/bithumb_bot/strategy/exit_rules.py`
* research backtest logic: `src/bithumb_bot/research/backtest_engine.py`
* research strategy routing: `src/bithumb_bot/research/strategy_registry.py`
* decision equivalence comparison: `src/bithumb_bot/decision_equivalence.py`
* profile / promotion / evidence flow: `src/bithumb_bot/profile_cli.py`, `src/bithumb_bot/approved_profile.py`
* docs and tests around research validation, approved profiles, promotion, runtime strategy, and evidence

The practical risk is that “the strategy validated in research” and “the strategy actually used in paper/live runtime” can drift while still appearing to be validated. The fix must move the repo toward a more reproducible, auditable, and operable automated trading experiment system.

Do not solve this by copying a few conditions from one file to another unless that is explicitly justified as a temporary compatibility adapter. Prefer design changes that prevent the entire class of research/runtime decision drift.

## Required design direction

Prioritize a structural fix.

The target architecture is:

```text
CandleSlice + StrategyProfile + FeeAuthoritySnapshot + PositionStateSnapshot
+ RuntimeClock/CandleBasis + RegimeSnapshot
        ↓
Shared Decision Kernel or Canonical Decision Adapter
        ↓
CanonicalDecisionEvent
        ↓
research execution simulator OR runtime OMS
```

The shared decision layer does not need to know about broker APIs, live order lifecycle, or actual fill handling. It must determine the canonical strategy decision: BUY intent, SELL exit intent, HOLD, and the complete reason/blocking context.

If introducing a full shared decision kernel is too invasive for one change, implement a concrete non-superficial first slice:

1. Define a `CanonicalDecisionEvent` representation in repo code.
2. Add adapters that export canonical decisions from runtime `SmaCrossStrategy.decide()` and from research backtest/replay logic.
3. Extend decision equivalence to compare canonical strategy/safety semantics, not only side/block fields.
4. Add golden equivalence tests that fail on known drift cases.
5. Document remaining gaps and make promotion/evidence boundaries explicit.

Do not leave the repo with only a prose recommendation.
