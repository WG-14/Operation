The repository must fail closed for production-bound candidates unless execution calibration evidence is present, hash-bound, semantically valid, and passing. A minimal fix that merely sets `execution_calibration_required=true` in one manifest is insufficient. The expected outcome is a repo-owned policy surface that makes production-bound calibration requirements explicit, testable, reproducible, and visible.

## Required design direction

Implement a central policy layer rather than scattering one-off checks.

Preferred shape:

Create or extend a module such as:

* `src/bithumb_bot/deployment_policy.py`, or
* `src/bithumb_bot/research/deployment_policy.py`, if research-local placement fits better.

The policy should express production-bound requirements in one place. Suggested concepts:

* `deployment_tier` or `promotion_target`
* allowed values such as:

  * `research_only`
  * `paper_candidate`
  * `live_dry_run_candidate`
  * `small_live_candidate`
* a function that returns required execution-calibration policy for a given target/profile mode
* fail-closed validation helpers with stable reason codes.

Production-bound tiers should require, at minimum:

* explicit `execution_model`, not only legacy `cost_model`
* `calibration_required=true`
* `calibration_strictness="fail"`
* calibration artifact content hash
* calibration artifact market and interval matching the manifest/profile
* calibration artifact fill reference policy matching execution timing when available
* calibration artifact execution reality level matching the expected policy when available
* `quality_gate_status="PASS"` when production-bound
* minimum sample count from existing calibration comparison policy
* no insufficient evidence
* no slippage/latency/partial-fill/unfilled/model-breach violation against the scenario assumptions.

Do not make `warn` strictness production-promotable. `warn` may remain acceptable for research-only diagnostics, but production-bound profile generation or profile promotion should reject it with an explicit reason code.

## Scope requirements

### P0: Central production-bound calibration policy

Add or reuse a central policy validator and wire it into the lifecycle at the earliest correct points.

At minimum, production-bound policy should be enforced before any artifact can be treated as eligible for `live_dry_run` or `small_live` profile transition.

Do not make research-only experimentation impossible. Research diagnostics may still run with legacy cost models, optional calibration, or warn-mode calibration, but such artifacts must not be accepted as production-bound evidence.

### P1: Calibration hash binding and reproducibility

Ensure calibration evidence is first-class and reproducible.

Specifically verify and, if needed, fix:

* combined calibration gates should expose top-level calibration artifact hash information, not only nested scenario gates
* promotion artifacts should bind calibration artifact hash consistently
* lineage should carry the same calibration artifact hash or an explicitly deterministic artifact hash set
* `research-reproduce` should detect calibration hash mismatch when calibration was required or bound
* if multiple scenarios reference calibration artifacts:

  * either require a single consistent calibration artifact for production-bound candidates, or
  * explicitly support and hash a deterministic sorted artifact hash set
* add stable reason codes for missing, inconsistent, hashless, or mismatched calibration artifacts.
