## Design priority

Prioritize a structural design change over symptom patches.

Do not implement this as:

* “randomly return `None` from `paper_execute()`”
* “sometimes halve the qty inside `paper_execute()`”
* “add a hardcoded failed-order branch without OMS lifecycle evidence”
* “mark partial fills as `FILLED` just because some trade was recorded”

Instead, introduce or refactor toward a common execution lifecycle abstraction that can support:

```text
Strategy signal
  -> order intent / sizing / submit plan
  -> execution adapter
  -> order lifecycle events
  -> OMS status transitions
  -> fill accounting
  -> reconcile / recovery / terminalization
```

The first implementation slice may be smaller than this full design, but it must move in that direction and not make later convergence harder.

## Required implementation direction

### 1. Preserve existing paper behavior by default

Current paper behavior using top-of-book, spread validation, slippage, guardrails, sizing, lot/dust handling, dedup, and ledger writes should continue to pass existing tests.

Introduce a default adapter or equivalent abstraction, for example:

```text
ImmediateTopOfBookPaperAdapter
```

This adapter should preserve the current behavior when stress execution is disabled.

### 2. Add a deterministic stress paper execution path

Add a stress-enabled paper execution path that reuses or adapts the existing research `StressExecutionModel` rather than inventing a separate nondeterministic implementation.

The stress path must support at least these deterministic scenarios:

* full fill
* partial fill
* order failure before fill accounting
* submit unknown / delayed fill if feasible in the first implementation slice
* latency evidence, even if actual scheduling is implemented later

Use deterministic seed/hash semantics. Do not use uncontrolled randomness.

Suggested configuration names may include:

```text
PAPER_EXECUTION_MODEL=immediate|stress
PAPER_EXECUTION_STRESS_SEED
PAPER_EXECUTION_LATENCY_MS
PAPER_EXECUTION_PARTIAL_FILL_RATE
PAPER_EXECUTION_PARTIAL_FILL_FRACTION
PAPER_EXECUTION_ORDER_FAILURE_RATE
```

If the repository already has different naming conventions, follow those conventions.

### 3. Apply lifecycle semantics, not just fill math

For a stress partial fill:

* create/order-record the requested qty
* apply only the filled qty to fills/trades/portfolio
* keep the order status as `PARTIAL`
* ensure `qty_filled < qty_req`
* preserve remaining qty evidence
* do not update intent dedup as terminal `FILLED`

For a stress failure:

* create an order or intent record if the failure occurs after intent/submission staging
* mark order/intent as `FAILED`
* do not create fills or trades
* include failure evidence in order events or another durable diagnostic location

For a submit unknown / delayed scenario if implemented:

* leave order in `SUBMIT_UNKNOWN`, `NEW`, or another appropriate open/recovery state according to existing OMS semantics
* ensure unresolved order gates can see it
* ensure duplicate intents do not silently create another order
* include replayable evidence showing why it is unresolved

### 4. Persist observability and replay evidence

Persist enough evidence to reproduce and audit the paper stress result. Prefer existing DB/event structures if possible; add schema migration only if necessary.

At minimum, preserve these values in a durable place such as `order_events.message`, `submit_evidence`, a JSON evidence column, or explicit columns if the schema already supports them:

```text
execution_model_name
execution_model_version
execution_model_params_hash
base_seed
derived_seed_hash
fill_status
requested_qty
filled_qty
remaining_qty
latency_ms
best_bid
best_ask
spread_bps
quote_source
quote_age_ms
execution_reality_level
```

If some fields are unavailable in paper runtime, record explicit `null`/`unknown` values rather than silently omitting the execution evidence.

### 5. Keep recovery and dedup semantics safe

Ensure the stress paper path respects existing OMS/dedup safety:

* `claim_order_intent_dedup`
* `update_order_intent_dedup`
* `OPEN_ORDER_STATUSES`
* terminal vs non-terminal status rules
* unresolved order gate behavior
* status transition validation where applicable

Do not release dedup for a `PARTIAL`, `SUBMIT_UNKNOWN`, `ACCOUNTING_PENDING`, or `RECOVERY_REQUIRED` order as if it were terminal.

### 6. Avoid breaking live trading paths

This task is primarily paper-runtime hardening. Do not make risky live-order behavioral changes unless they are small, well-tested, and necessary for shared lifecycle utilities.

If you introduce shared lifecycle helpers, prove they preserve existing live tests.

### 7. Document live-dry-run chaos gap without over-scoping

If adding a full `LiveDryRunChaosBroker` is too large for this change, add a focused design note or TODO-backed test scaffolding that makes the gap explicit.

The design should call for deterministic dry-run scenarios such as:

```text
submit_timeout_then_reconcile
broker_reject_under_min_total
partial_fill_then_fee_pending
order_not_ready_then_recovery_required
```

Do not claim the live dry-run gap is solved unless code and tests actually exercise it.


Keep this lightweight. Do not over-engineer a full framework if a smaller adapter plus tests can establish the direction.

When adapting `StressExecutionModel`, map paper runtime data into `ExecutionRequest` using:

* signal ts
* decision ts
* side
* reference price
* requested qty or notional
* fee rate
* best bid/ask
* spread bps
* quote source if available
* latency settings
* seed derivation inputs that include client order id or intent key only if that preserves deterministic replay expectations

Be careful: using a UUID client order id in seed derivation may break deterministic replay. Prefer stable inputs such as intent key, signal ts, side, symbol, strategy context, requested qty, model params hash, and explicit base seed.
